# Route Ownership

This file documents which repo owns each public route at `engineering.diversio.com`.

## Why this split exists

The site renders from one repo (`engineering-website`) but some content lives in another (`agent-skills-marketplace`). This is deliberate:

- **Agentic Tools content is source-owned by the tools repo.** Skill definitions, package READMEs, and catalog metadata are maintained alongside the tools they describe. Copying them into this repo would create drift.
- **The broader engineering site is editorially independent.** How-we-work, systems notes, blog posts, and community pages are maintained here without coupling to tool releases.
- **One deployment, one domain.** Visitors see a single cohesive site at `engineering.diversio.com`. The split is a build-time concern, not a user-facing one.

**Column meanings**:
- **Rendered by**: which repo builds and deploys this page.
- **Source of truth**: which repo owns the canonical content. For Agentic Tools routes the content lives in `agent-skills-marketplace` and is resolved at build time via `site-docs.ts` extraction.

## Source resolution

The new repo resolves Agentic Tools content from an `agent-skills-marketplace` checkout at build time. Resolution order:

1. `AGENT_SKILLS_REPO_DIR` environment variable
2. Current git repo root (when it contains `plugins/` and `pi-packages/`)
3. `../` (parent directory)
4. `../agent-skills-marketplace` (sibling checkout)
5. `../vendor/agent-skills-marketplace` (vendored sibling)
6. `vendor/agent-skills-marketplace` (vendored in-project)

The same resolution logic is used by both the Astro build (`src/data/site-docs.ts`) and OG image generation (`scripts/generate-og-images.py`). Both check `AGENT_SKILLS_REPO_DIR` first, then sibling and vendored checkouts. The exact candidate order differs slightly between the two but both resolve correctly for the standard sibling checkout pattern.

## Key boundaries

- **`agent-skills-marketplace`** is the canonical source for all tool docs, skill definitions, package READMEs, and catalog metadata. Edits to these must happen in ASM, not in `engineering-website`.
- **`engineering-website`** owns all page rendering, shared layouts, components, styles, blog content, and the production deployment. Edits to page structure, site identity, or editorial content happen here.
- **`marketplace.json`** in this repo is a build artifact (`src/data/marketplace.json`). It is copied from the ASM checkout before each build. It is not committed source. Do not hand-edit it. The canonical source is ASM's `website/src/data/marketplace.json`.

## Workflow ownership

Who owns each CI/deploy concern after the split.

| Concern | Owner | Notes |
|---|---|---|
| Build validation (PR gate) | engineering-website | `.github/workflows/validate-website.yml` |
| Preview deploys (PR review) | engineering-website | `.github/workflows/deploy-website-cloudflare-pages.yml` (same-repo PRs only) |
| Production deploys (push to main) | engineering-website | Same workflow, production job, uses `agent-skills-source.ref` lock file |
| Production deploys (ASM content update) | engineering-website (triggered by ASM) | `repository_dispatch` from ASM with exact SHA overrides lock file |
| Trigger dispatch (ASM main merge) | agent-skills-marketplace | `.github/workflows/trigger-engineering-website-deploy.yml` (main-only) |
| ASM-local build validation | agent-skills-marketplace | `.github/workflows/validate-website.yml` (read-only, does not deploy) |
| Old ASM deploy workflow | RETIRED | `deploy-website-cloudflare-pages.yml` has `if: false` on all jobs |
| Cloudflare Pages project | engineering-website | Project `diversio-engineering`, credentials in repo secrets |
| Domain (engineering.diversio.com) | Cloudflare Pages | Configured in Cloudflare dashboard, not in repo |
